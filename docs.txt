  1. WHAT THIS PROJECT IS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

wolfmusicapi is a full-stack REST API platform and web hub hosted at
apis.xwolf.space. It serves two purposes:

  1. A public API server — offering 100+ endpoints across music, video
     downloads, social media, image search, AI, OSINT/stalker tools,
     sports, movies, converters, image/text effects, and more.

  2. A web landing page — a neon-themed dark UI that showcases the APIs,
     lists every available endpoint, and links to related projects.

The project runs as a single unified Node.js process on a VPS (port 5000).
Express handles both the API routes and serves the built React frontend. In
development, Vite serves the frontend with hot-module reload. In production,
the pre-built static files are served directly by Express.

  2. TECH STACK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Backend    : Node.js + Express 5 (TypeScript via tsx / esbuild)
  Frontend   : React 18 + Vite + TypeScript
  UI Library : shadcn/ui (Radix UI primitives + Tailwind CSS)
  Routing    : wouter (lightweight client-side router)
  Data Fetch : TanStack Query v5
  Fonts      : Orbitron (headings), JetBrains Mono (code/mono)
  Process Mgr: PM2 (production, VPS)
  Build      : Custom script/build.ts — runs Vite, then esbuild
  Deployment : VPS via SSH + SCP, or deploy.sh on the server itself
  Version Ctl: GitHub (github.com/nk-apex/wolfmusicapi)

wolfmusicapi/
│
├── admin-settings.json          ← LIVE runtime config. Stores: admin password,
│                                   GitHub URL, project cards (repo links), and
│                                   the IP blocklist. Written by the server at
│                                   runtime when settings change via the admin
│                                   panel. NOT committed to git (.gitignore) to
│                                   protect live credentials and configuration.
│
├── deploy.sh                    ← VPS deployment script. Run this on the VPS
│                                   after pushing new code to GitHub. It:
│                                     1. Backs up admin-settings.json to /tmp
│                                     2. Removes it so git pull can run cleanly
│                                     3. git pull origin main
│                                     4. Merges live settings back in using Node
│                                        (preserves password, blocklist, cards,
│                                        but picks up new fields from git)
│                                     5. npm run build
│                                     6. pm2 restart wolfmusicapi
│                                   This prevents settings loss on every deploy.
│

│                                     npm run dev   → starts dev server (tsx)

│ ── SERVER ─────────────────────────────────────────────────────────────────
│
├── server/
│   │
│   ├── index.ts                 ← MAIN ENTRY POINT. This is what Node.js runs.
│   │                               Responsibilities:
│   │                               - Creates the Express app and HTTP server
│   │                               - Sets trust proxy (for real IPs behind Nginx)
│   │                               - Applies the full middleware stack in order:
│   │                                   1. securityHeaders()      → Helmet HTTP headers
│   │                                   2. ipBlocklistGuard       → blocks banned IPs
│   │                                   3. globalLimiter          → 300 req/15min flood guard
│   │                                   4. botBlocker             → blocks scanner User-Agents
│   │                                   5. blockDirectSourceAccess→ hides .ts/.tsx source files
│   │                                   6. antiClone              → protects endpoint list
│   │                                   7. express.json()         → parses JSON request bodies
│   │                                   8. express.urlencoded()   → parses form data
│   │                                   9. URL decode validator   → rejects malformed URLs
│   │                                  10. Request logger         → logs all /api calls
│   │                               - Calls registerRoutes() to mount all API routes
│   │                               - In production: serves the built frontend from dist/
│   │                               - In development: mounts Vite as middleware (HMR)
│   │                               - Auto-updates yt-dlp binary on startup
│   │                               - Listens on port 5000 (or $PORT env var)
│   │
│   ├── routes.ts                ← ALL API ROUTES. The largest file (~3400 lines).
│   │                               Contains:
│   │                               - Activity tracking middleware — records every
│   │                                 API request into a 300-entry ring buffer (the
│   │                                 "Live Log" tab reads this buffer)
│   │                               - Daily stats tracking — counters that reset at
│   │                                 midnight (today's requests, top API today)
│   │                               - Per-IP request tracking — feeds the Security tab
│   │                               - adminLimiter applied to all /api/admin/* routes
│   │                               - heavyLimiter applied to all /download/* routes
│   │                               - All public API endpoints:
│   │                                   /api/search          → YouTube/music search
│   │                                   /api/download/*      → audio/video downloads
│   │                                   /api/tiktok          → TikTok downloader
│   │                                   /api/instagram       → Instagram downloader
│   │                                   /api/facebook        → Facebook downloader
│   │                                   /api/twitter         → Twitter/X downloader
│   │                                   /api/snapchat        → Snapchat downloader
│   │                                   /api/spotify/*       → Spotify search/download
│   │                                   /api/shazam          → Shazam recognition
│   │                                   /api/yandex/*        → Yandex image/video search
│   │                                   /api/stalk/*         → OSINT/lookup endpoints
│   │                                   /api/sports/*        → Sports scores/fixtures
│   │                                   /api/movie/*         → Movie/TV info (xcasper)
│   │                                   /api/ai/*            → AI chat endpoints
│   │                                   /api/tools/*         → QR, hash, UUID, etc.
│   │                                   /api/image-effects/* → Ephoto360, PhotoFunia
│   │                                   /api/text-effects/*  → TextPro effects
│   │                                   /api/audio-effects/* → Audio processing
│   │                                   /api/convert/*       → Media format conversion
│   │                                   /api/fun/*           → Jokes, memes, facts
│   │                                   /api/anime           → Anime images
│   │                                   /api/url/shorten     → URL shortener
│   │                               - All admin endpoints (see Section 6)
│   │                               - /api/proxy — streams remote media URLs
│   │                               - /api/tempfile/:id — serves yt-dlp temp files
│   │                               - /api/config/cards — public endpoint for homepage
│   │                               - /api/endpoints/list — full endpoint catalogue
│   │
│   ├── security.ts              ← SECURITY LAYER. Exports all middleware and rate
│   │                               limiters used across index.ts and routes.ts:
│   │                               - Per-IP tracking: trackIpRequest(), getTopIpsToday(),
│   │                                 getSecurityStats() — feeds the Security tab
│   │                               - botBlocker() — blocks by User-Agent signature:
│   │                                   sqlmap, nikto, nmap, scrapy, nuclei, masscan,
│   │                                   dirbuster, gobuster, ffuf, wfuzz, hydra,
│   │                                   burpsuite, acunetix, nessus, openvas
│   │                                   Also blocks empty UA on /api/* paths → HTTP 403
│   │                               - ipBlocklistGuard() — reads blocklist from
│   │                                   admin-settings.json and returns 403 for banned IPs
│   │                               - 5 rate limiters (express-rate-limit):
│   │                                   globalLimiter  → 300 req / 15 min / IP
│   │                                   loginLimiter   → 5 attempts / 15 min / IP
│   │                                   apiLimiter     → 80 req / min / IP
│   │                                   heavyLimiter   → 20 req / min / IP
│   │                                   adminLimiter   → 30 req / 5 min / IP
│   │                               - securityHeaders() — Helmet config (CSP, referrer
│   │                                   policy, frame options, CORP, COOP)
│   │                               - antiClone() — blocks non-browser access to
│   │                                   /api/endpoints/list to prevent API cloning
│   │                               - responseFingerprint() — appends _powered and _ts
│   │                                   fields to every JSON response
│   │                               - blockDirectSourceAccess() — returns 404 for
│   │                                   requests to .ts, .tsx, .map, .env file paths
│   │                                   and /server/, /lib/, /shared/, /.git/ paths
│   │
│   ├── admin-settings.ts        ← SETTINGS PERSISTENCE MODULE.
│   │                               - Defines AdminSettings interface:
│   │                                   { password, githubUrl, repoCards[], ipBlocklist[] }
│   │                               - Reads and writes admin-settings.json on disk
│   │                               - In-memory cache (_settings) for fast reads
│   │                               - loadSettings() — called once at server startup
│   │                               - getSettings() — returns the cached settings
│   │                               - saveSettings() — writes to disk + updates cache
│   │                               - DEFAULT_SETTINGS — fallback if file is missing
│   │
│   ├── scraper.ts               ← RE-EXPORT SHIM. A one-line file that re-exports
│   │                               everything from lib/scraper.ts so that server/
│   │                               routes.ts can import from a clean relative path.
│   │
│   ├── ai-routes.ts             ← AI CHAT ROUTES. Registers four AI endpoints:
│   │                               - POST /api/ai/chat         → OpenAI GPT completion
│   │                               - POST /api/ai/groq         → Groq fast inference
│   │                               - POST /api/ai/gemini       → Google Gemini
│   │                               - POST /api/ai/chateverywhere → proxy to chateverywhere.app

│   ├── static.ts                ← STATIC FILE SERVING. In production, serves the

  4. HOW A REQUEST FLOWS THROUGH THE SYSTEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Example: GET /api/download/tiktok?url=https://tiktok.com/...

  1. Request arrives at the VPS. Nginx terminates HTTPS and forwards to port 5000.

  2. Express processes the middleware stack in server/index.ts:
       a. ipBlocklistGuard   → checks IP against admin-settings.json blocklist
       b. globalLimiter      → checks if IP has exceeded 300 req/15min
       c. botBlocker         → checks User-Agent against scanner signatures
       d. blockDirectSourceAccess → passes (is an /api/ path, not a source file)
       e. antiClone          → passes (not the endpoints list route)
       f. express.json()     → parses body (GET has no body)
       g. URL validator      → validates the URL encoding is safe
       h. Request logger     → records start time

  3. routes.ts middleware fires:
       a. Activity tracker   → records the request IP, starts timing
       b. heavyLimiter       → checks /api/download/* limit (20/min)

  4. The route handler executes:
       - Extracts url from query params
       - Calls downloadTikTok(url) from lib/downloaders/tiktok.ts
       - tiktok.ts scrapes ssstik.io with the URL
       - Returns: { success, videoUrl, noWatermarkUrl, audioUrl, author, stats }

  5. Response is sent back:
       - responseFingerprint() adds _powered and _ts fields to the JSON
       - Activity tracker records: endpoint, HTTP status, duration
       - Daily counter and per-IP tracker are incremented

  6. Admin panel reflects it within seconds:
       - Overview: dailyTotalRequests counter goes up
       - Live Log: this request appears within 5 seconds
       - Security: this IP's hit count increases
       - Top APIs Today: endpoint ranking updates

  5. HOW THE MULTI-PROVIDER MUSIC DOWNLOAD CHAIN WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

When a user requests /api/download/mp3?url=<youtube_url>, lib/scraper.ts runs
getDownloadInfo(url, "mp3") which tries providers in this order:

  1. FabDL     → Fast, reliable. Sends URL to the FabDL service API.
  2. Invidious → Open-source YouTube frontend that exposes audio streams.
  3. Cobalt    → cobalt.tools — open-source download service.
  4. Piped     → Another YouTube frontend with direct stream access.
  5. Y2Mate   → y2mate.com API.
  6. YTDown   → ytdown.to — PHP session + CDN polling (see docs below).
  7. yt-dlp   → Final fallback. Spawns the yt-dlp binary, downloads to
                 /tmp/wolfapi_dl/<uuid>.mp3, serves as a stream, then
                 auto-deletes the file after 10 minutes.

How provider health works:
  Each provider has a success/failure score tracked in memory. After enough
  consecutive failures, the provider is marked DOWN and skipped for future
  requests. This is automatic — if cobalt.tools goes down at 3am, the system
  just routes around it without any manual intervention. Providers can be
  manually reset to UP status from the admin Providers tab.

  6. ADMIN PANEL — HOW IT WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

URL: https://apis.xwolf.space/admin

Authentication model:
  - Login form POSTs to POST /api/admin/login with { password }
  - On success, password is stored in browser sessionStorage (tab-scoped)
  - Every subsequent admin API call sends it as: x-admin-password header
  - Server validates against admin-settings.json on every single request
  - No JWT, no session cookie — the password IS the token
  - Login is rate-limited: 5 attempts per 15 minutes per IP (brute-force guard)

Admin API endpoints (all require x-admin-password header):

  GET  /api/admin/stats          → traffic stats, top endpoints, daily counts
  GET  /api/admin/logs           → last N request log entries (the Live Log)
  GET  /api/admin/settings       → current githubUrl and repoCards config
  POST /api/admin/settings       → update githubUrl or repoCards
  POST /api/admin/change-password→ change the admin password (written to disk)
  GET  /api/admin/security       → IP tracking stats, blocklist, top IPs today
  POST /api/admin/block-ip       → add IP to blocklist (takes effect immediately)
  POST /api/admin/unblock-ip     → remove IP from blocklist
  GET  /api/admin/provider-health→ UP/DOWN status of all download providers
  POST /api/admin/reset-health   → reset a specific provider back to UP
  GET  /api/admin/update-ytdlp   → triggers yt-dlp self-update on the server
  GET  /api/admin/reload-cookies → reloads YouTube cookies from disk

Public endpoints (no auth required):
  GET  /api/config/cards         → returns githubUrl + repoCards for the homepage
  GET  /api/endpoints/list       → returns the full API endpoint catalogue
                                   (antiClone middleware restricts browser-only access)

  7. SECURITY SYSTEM — HOW IT WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Five layers of protection run on every incoming request:

  LAYER 1 — IP Blocklist (ipBlocklistGuard)
    Reads the blocklist from admin-settings.json (in-memory cache, fast).
    Returns HTTP 403 immediately if the request IP is on the list.
    Blocking and unblocking from the admin panel take effect instantly —
    no server restart needed.

  LAYER 2 — Flood Protection (globalLimiter)
    300 requests per 15 minutes per IP across all /api/ and /download/ paths.
    Returns HTTP 429 with a Retry-After header if the limit is exceeded.
    Uses express-rate-limit's default IP resolution which correctly handles
    IPv4, IPv6, and IPv4-mapped IPv6 addresses.

  LAYER 3 — Bot / Scanner Detection (botBlocker)
    Inspects the User-Agent header against a list of known attack tool strings.
    Also blocks requests with an empty User-Agent on /api/ paths.
    Blocked tools: sqlmap, nikto, nmap, scrapy, nuclei, masscan, dirbuster,
    gobuster, ffuf, wfuzz, hydra, burpsuite, acunetix, nessus, openvas.
    Returns HTTP 403 for any match.

  LAYER 4 — Endpoint-Specific Rate Limits
    adminLimiter  → /api/admin/*        : 30 req / 5 min / IP
    loginLimiter  → /api/admin/login    : 5 req / 15 min / IP (brute-force)
    heavyLimiter  → /download/* and /api/download/* : 20 req / min / IP
    apiLimiter    → available for individual endpoints if needed

  LAYER 5 — Security Headers (Helmet)
    Content-Security-Policy: restricts script, style, image, and font origins
    Referrer-Policy: strict-origin-when-cross-origin
    Cross-Origin-Resource-Policy: cross-origin (allows API calls from anywhere)
    Blocks iframe embedding (frameAncestors: self), <object> tags, base-URI abuse

All counters and IP tracking are in-memory and reset on server restart.
The IP blocklist is the only security state persisted to disk (admin-settings.json).
Daily stats and IP hit counts reset automatically at midnight.

  10. ADDING A NEW API ENDPOINT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Step 1: Write the logic
    Create lib/downloaders/<name>.ts with the function that fetches and
    processes data for the new feature.

  Step 2: Register the route
    In server/routes.ts, import your function and add:
      app.get("/api/<path>", async (req, res) => { ... });
    For expensive operations, add the heavyLimiter middleware:
      app.get("/api/<path>", heavyLimiter, async (req, res) => { ... });

  Step 3: Document it
    Add an entry to the allEndpoints array in shared/schema.ts with:
      path, method, description, parameters, category
    This automatically makes it appear on the homepage endpoint list.

  Step 4: Build and test
    npm run build
    curl https://apis.xwolf.space/api/<path>

  Step 5: Deploy
    git push, then on VPS: bash /var/www/wolfmusicapi/deploy.sh

  11. KEY DESIGN DECISIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Single port, single process
    Express serves both the REST API and the static React app from port 5000.
    No separate frontend server in production. Simplifies VPS setup and
    eliminates CORS issues entirely since everything is the same origin.

  No database required for core features
    All runtime state (stats, logs, settings, provider health) lives in memory
    or in a single JSON file. Fast reads, zero infrastructure dependencies,
    easy to understand and debug.

  Multi-provider fallback chain for downloads
    Instead of depending on one external service, the system tries 7 providers
    in sequence and falls back automatically. If any service goes down, users
    never notice. The final fallback (yt-dlp) runs locally so the API always
    has a way to fulfill the request.

  Runtime-editable settings without restart
    The admin password, homepage cards, and IP blocklist all live in
    admin-settings.json and are hot-reloaded without restarting Node. The
    in-memory cache is updated on every saveSettings() call.

  Security in middleware, not per-route
    All five protection layers are applied before any route handler runs.
    New endpoints added in the future automatically inherit full security
    protection without any extra code.

  Self-documenting API via shared schema
    Both the frontend and backend import from shared/schema.ts. The endpoint
    list on the homepage is always 100% in sync with what is actually deployed —
    there is no separate documentation file to maintain or forget to update.

WolfAPIs v4.0 — Site Scraping Research Notes
=============================================
Written by: reverse-engineering + network analysis (no browser automation used)
Purpose: Educational documentation of how each music/video download site was
         reverse-engineered so you can replicate the technique yourself.

═══════════════════════════════════════════════════════════════════════════════
HOW TO REVERSE-ENGINEER A DOWNLOAD SITE (GENERAL METHOD)
═══════════════════════════════════════════════════════════════════════════════

The same 4-step workflow was used for every site below:

  1. Open the site in Chrome DevTools → Network tab (filter: XHR / Fetch)
  2. Paste a YouTube URL and click "Download"
  3. Watch which requests fire, in order
  4. Right-click a request → "Copy as cURL" → replay from curl/Node.js

Key things to look for in the Network tab:
  • The "Initiator" column tells you which JS file started each request
  • Sort by time — the download-triggering request is usually 1-3 steps in
  • Check "Cookies" tab for session tokens the site creates before the API call
  • If a request returns HTML instead of JSON, look for inline <script> tags
    that call a secondary API endpoint

Once you have the raw cURL, strip it to the minimum working set of headers.
Usually only User-Agent, Origin, Referer, and any cookie are required.
The rest (Sec-Fetch-*, DNT, etc.) can be omitted.

To find hidden API endpoints without DevTools:
  • View page source → search for "fetch(", "$.post(", "$.getJSON(", "axios."
  • Search for ".php", "/api/", "/convert", "/download" in the JS
  • In Nuxt/React apps: download the main bundle JS file and grep it for the
    apiBase / baseURL config object — this reveals the real backend domain

SITE 1: ytdown.to  ✅  FULLY IMPLEMENTED (Provider "ytdown")
═══════════════════════════════════════════════════════════════════════════════

URL: https://app.ytdown.to/en23/

How it was discovered:
  Network tab showed two requests on form submit:
    1. The page itself (GET /en23/) sets a PHPSESSID session cookie
    2. A POST to /proxy.php with the YouTube URL in the body

Why it's useful:
  The server returns download URLs on ytdown.to's own CDN (s*.ytcontent.com),
  NOT on YouTube's servers. This completely bypasses YouTube's IP blocking of
  VPS/datacenter IPs. Anyone can download from the CDN URL.

Full API flow:

  STEP 1 — Get PHP session cookie:
    GET https://app.ytdown.to/en23/
    Headers: standard browser headers
    → Response sets: PHPSESSID=<hex string>  (via Set-Cookie header)
    → The session is valid for ~25 minutes. Cache it and reuse.

  STEP 2 — Request download info:
    POST https://app.ytdown.to/proxy.php
    Headers:
      Content-Type: application/x-www-form-urlencoded
      Origin: https://app.ytdown.to
      Referer: https://app.ytdown.to/en23/
      X-Requested-With: XMLHttpRequest
      Cookie: PHPSESSID=<value from step 1>
    Body (URL-encoded):
      url=https://www.youtube.com/watch?v=dQw4w9WgXcQ

    Response (JSON):
    {
      "api": {
        "title": "Never Gonna Give You Up",
        "mediaItems": [
          {
            "type": "Video",
            "mediaQuality": "FHD",       // FHD=1080p, HD=720p, SD=480/360/240/144
            "mediaExtension": "MP4",
            "mediaFileSize": "83.70 MB",
            "mediaDuration": "3:32",
            "mediaUrl": "https://s19.ytcontent.com/v5/video/dQw4w9WgXcQ/TOKEN/1080p"
          },
          {
            "type": "Audio",
            "mediaQuality": "128K",
            "mediaExtension": "MP3",
            "mediaFileSize": "7.20 MB",
            "mediaDuration": "3:32",
            "mediaUrl": "https://s19.ytcontent.com/v5/audio/dQw4w9WgXcQ/TOKEN/128k"
          }
        ]
      }
    }

    Note: mediaUrl is a status/polling endpoint, NOT the direct download URL.

  STEP 3 — Poll the CDN status URL:
    GET https://s19.ytcontent.com/v5/video/dQw4w9WgXcQ/TOKEN/1080p

    When completed:
    { "status": "completed", "fileUrl": "https://s19.ytcontent.com/v5/download/..." }

    When pending:
    { "status": "queued", "percent": "0%", "position": 1 }
    → Wait ~3 seconds and poll again (max 15 attempts = ~45 seconds)

  STEP 4 — fileUrl is the real download link (signed, expires shortly).

Implementation notes:
  • Session cached in memory for 25 minutes to avoid extra round-trips
  • On non-OK response from proxy.php → invalidate session (triggers re-fetch)

SCRAPING TECHNIQUES LEARNED
═══════════════════════════════════════════════════════════════════════════════

1. PHP SESSION GATING — Many PHP download sites require a PHPSESSID cookie
   before responding to API calls. Issue a GET to any page first, cache the
   cookie, and reuse it across requests for ~25 minutes.

2. X-Requested-With: XMLHttpRequest — Always include this header when the
   Network tab shows it on the original request. Sites use it to distinguish
   AJAX from regular page loads.

3. NUXT.JS APP DETECTION — If the page source contains window.__NUXT__, look
   for window.__NUXT__.config.public.apiBase to find the real backend URL.
   Route definitions are in the _nuxt/*.js bundle files.

4. CDN POLLING PATTERN — ytdown.to uses a two-level URL pattern:
   Level 1 (proxy.php): returns per-quality status endpoint URLs
   Level 2 (CDN URL): returns { status, fileUrl } — poll until "completed"
   This is common in async video conversion services. fileUrl expires quickly.

5. CLOUDFLARE DETECTION — Headers cf-mitigated: challenge and Server: cloudflare
   mean CF Bot Fight Mode is active. Plain HTTP clients cannot bypass this.

6. IFRAME WIDGET PATTERN — Sites embed download widgets as third-party iframes.
   To scrape these, you must either reverse-engineer the iframe's backend API
   from its JavaScript bundle, or use Puppeteer to intercept requests inside
   the iframe during real browser execution.
